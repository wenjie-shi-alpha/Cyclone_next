学术会议汇报版 · Phase 2 诊断监督

模型已经很准，
为什么专家仍然更好？

这份汇报要讲清楚一个完整故事。第一步，我们先证明专家预报的优势依然真实存在；第二步，我们说明专家的修正不是偶然， 而是落在稳定、可识别的误差路径上；第三步，我们在本项目中设计一套 可执行、可验证、可集成 的结构化中间接口，让 LLM 学习专家如何修正 guidance， 并检验这种“学会推理过程”的能力能否真正改善下游 forecast。

背景证据 `AIvsNWP` 显示 `OFCL` 平均轨迹误差 `92.9 km`，显著优于 `GFS 159.3 km` 与 `AI_FOUR 282.4 km`
机制证据 `MOSNet` 显示专家会沿 6 类稳定误差路径做系统补偿，而不是随机修正
本项目答案 当前最有效的是 13 字段 `slot-turn-correction` 接口，而不是更宽、更抽象的 schema
当前位置 confirmed mainline 在 held-out `n=1454` 上把轨迹误差从 `181.91 km` 降到 `145.58 km`
专家优势已证实 专家修正机制已证实 结构化过程学习已证实 v3 full confirm 已完成

一页摘要

当前最稳妥的结论不是“LLM 已经学会像专家写解释”，而是： LLM 已经学会了一部分专家使用的结构化判断状态，并且这部分判断状态可以被渲染成真正更好的轨迹预报。 因此，本阶段的贡献是把“专家过程是否可学、是否有用”从猜想推进到数据可验证的层面。

Problem Framing

问题抛出：模型已经很强，为什么还要研究专家过程

因为“模型已经能给出预报”并不等于“模型已经吸收了专家决策层的价值”。 从更广的业务比较到本项目自己的正式测试，专家官方预报都仍然保有明显优势，尤其体现在尾部风险控制与整体稳定性上。

Why Experts Matter

专家修正不是偶然拍脑袋，而是对可识别误差路径的系统补偿

`/root/MOSNet` 的 pathway 分析给了这一层证据：专家并不是在“随机修一点”，而是在一些稳定可识别的误差路径上， 长期、方向一致地做补偿。

这对本项目意味着什么

  • 专家优势不是抽象的“经验”，而是落在可命名、可统计、可验证的物理路径上。
  • 因此 Phase 2 最自然的建模目标，不是直接模仿 OFCL 文本，而是先恢复这些中间判断状态。
  • 这也解释了为什么 richer schema 未必有效：有效接口必须足够接近可执行修正，而不是停留在抽象描述层。
  • 我们当前的 `slot-locked correction interface`，本质上就是把“专家如何修正 guidance”压缩为一个 LLM 可学的结构化接口。
Storyline

完整研究故事：问题、选择与当前答案

这项研究围绕一个清晰问题展开：结构化过程监督是否能帮助模型学习专家的决策过程，并进一步改善轨迹预报。

故事起点：只有 forecast 结果还不够

第一阶段我们已经能训练出直接输出严格 forecast table 的模型，但这只能回答“模型能否拟合输出结果”， 不能回答“模型是否学到了专家在模式分歧、转向风险和轨迹修正上的判断过程”。如果没有这一步， 我们很难知道模型是在做可解释的决策，还是只是在记忆某种输出分布。

核心假设：先学结构化过程，再谈解释文本

因此二阶段不直接训练长文本 reasoning，而是把专家决策过程压缩成可验证的结构化 diagnostics / decision states。 只有当这套中间接口在 oracle 注入时能改善预报，且 predicted diagnostics 也能恢复一部分增益时， 我们才认为模型真的学到了一部分“如何做判断”。

当前答案：学到了“可执行的中间判断”

当前结果表明，最有效的不是更宽、更抽象的 schema，而是更窄、更可执行的 slot-locked correction interface。 这条线已经在 held-out 上完成确认，并成为当前推荐主线；剩余问题已收敛到少数弱字段，而不是整体方案仍不成立。

术语 1：Baseline 是什么

中文可表述为
“基线模型”或“直接预报参考线”。它表示如果我们完全不引入二阶段结构化过程监督，只做标准 forecast，会得到什么水平。
正式含义
Baseline 是第一阶段冻结的“直接做预报”的参考模型，不注入任何二阶段诊断接口，用作所有后续对照。
当前默认 baseline
`baseline_forecast_sft_v2`。它代表“只根据共享证据底座直接给出 forecast table”的标准做法。
为什么重要
如果二阶段方法不能稳定优于这个 baseline，就不能说过程监督真的创造了价值。

术语 2：Oracle 是什么

中文可表述为
“理想诊断注入”或“上限注入实验”。它不是实际部署方式，而是用来回答“如果中间接口完全正确，这套接口本身值不值得学”。
正式含义
Oracle 指“不让模型自己预测诊断结果”，而是把规则构造或近似真实的正确诊断信号直接注入 forecast 输入。
回答的问题
它回答的不是“模型学会了吗”，而是“这类接口本身有没有价值”。
为什么必须先做
如果 oracle 自己都不能改善 forecast，继续训练 predicted diagnostics 就没有意义。

术语 3：Predicted / Mainline / Control

Predicted 的中文含义
“模型先预测中间判断，再把该判断接入 forecast”。它对应我们真正关心的“模型是否学会专家过程”。
Predicted
先训练一个 diagnostic adapter 从输入证据预测结构化诊断，再把它注入 forecast 模型，检验模型是否学会这套过程。
Mainline 的中文含义
“当前主线方案”或“当前推荐方案”。它是现阶段最值得继续投入计算资源和分析精力的方案。
Confirmed mainline
在正式 compare 和更大 held-out confirm 上都过线的当前推荐方案。
Control 的中文含义
“稳定对照线”。它保留为参照物，用来防止我们被单次更优结果误导。
Control
被保留作为稳定性参照或消融对照的旧主线，例如 `slot_correction v1 + baseline intensity + scale 1.20`。

这个任务的输入和输出，实际长什么样

这里展示的不是抽象 schema，而是真实样本。可以看到我们并不是让模型“凭空思考”， 而是把观测、环境诊断和 guidance 组织成一个结构化提示，再要求模型输出严格可解析的结果。

共享输入证据

观测 + 环境 + guidance

所有阶段共享同一证据底座，包括当前位置、环境诊断、卫星/侦察观测和 ATCF / HRES guidance。

Phase 2 输出

13 字段 diagnostic JSON

先让模型输出可执行的中间判断状态，包括 1 个 `turning_signal` 与 12 个 `slot_1..slot_6` 的 lat/lon 修正桶。

渲染与集成

slot-locked renderer

再把这些结构化判断渲染成对已有 guidance 的受约束修正，不允许改时槽，不允许顺带改 intensity。

最终输出

严格 forecast table

最终仍然回到严格可解析的官方预报表，这样才能和 Phase 1 baseline 与专家官方预报直接比较。

Protocol

实验设计：为什么这些结论可信

结论的可信度来自严格的阶段划分、清晰的因果问题拆解，以及每一阶段都必须满足的对照与 go/no-go 条件。

Phase 2A：接口验证

先做 oracle compare，只问一个问题：如果把某种结构化决策状态直接交给 forecast 模型， 它能不能显著降低 `track_error_km` 和 `mean_track_diff_vs_official_km`。

Phase 2B：预测恢复

只有当接口本身被证明有效，才训练 predicted diagnostic adapter。 这一步回答的是：模型能否从同样的输入证据中恢复这套过程信息，并把一部分 oracle 增益带回 forecast。

Phase 2C：窄扩展，不做泛化发散

只有在 2A 和 2B 都通过后，才允许扩展 schema。并且我们已经用负结果证明： 当前阶段不能简单扩成 richer schema，必须沿“窄而可执行”的接口继续推进。

这个设计为什么有方法学价值

  • 它把“接口有没有价值”和“模型有没有学会接口”这两个问题拆开了，因此因果关系更清楚。
  • 它避免把 standalone 分类指标误当成真实 forecast 改善。
  • 它允许负结果直接产生方法学收敛，而不是继续无界扩展 schema。
Executive Meaning

这些结果的阶段性含义

在完成数据与方法说明后，最重要的是明确：当前结果已经回答了哪些问题，又把剩余问题收缩到了什么范围。

1. 不是“diagnostics 可能有用”

我们已经证明，某类结构化 diagnostics 不只是让标签更好看，而是可以真实降低 forecast track 主指标。 关键转折点是 `oracle slot_correction` 首次显著通过，说明接口本身是有效的。

2. 正确方向不是更宽 schema

`track_core v1` 这类 richer schema 已被证明不是当前正确主线。真正有效的是更窄、可执行、slot-locked 的轨迹修正接口， 后续扩展也必须沿 executable interface 这条线走。

3. 模型不只是在“直接给答案”

当前主线不仅 forecast 指标改善，而且 standalone contract 已修稳，`turning_signal` 的 exact accuracy 与 macro-F1 都显著提升。 这说明模型学到的是可执行的中间判断状态，而不只是表面输出。

4. 剩余问题已经收窄

当前 blocker 不再是 JSON parseability，也不是 generic redesign，而是 12 个 slot bucket 字段中最弱的语义分辨率， 尤其是 `lat bucket` 仍未真正修好。

Research Logic

我们现在这条线的核心逻辑

这项工作不是把长文本 reasoning 直接塞回 forecast 主任务，而是先验证：模型能否学会专家使用的结构化判断节点， 再看这些判断节点能否稳定改善下游 forecast。

长期链路

输入 → 诊断 → 预报 → 解释

项目长期目标是三段式链路：先输出结构化 diagnostics，再生成严格 forecast，最后才生成 explanation。 这意味着 reasoning 是解释层，不是当前主决策层。

为什么先不做自由文本 reasoning

先证明结构化中间接口有价值

reasoning 文本覆盖率低、自动评估难、容易破坏严格 forecast 格式，因此当前阶段的更稳做法是先学习结构化 decision state， 再把 explanation 放到后续阶段。

当前定义的“学会推理”

学会结构化判断，而不是学会写长文

这里的推理，不等于生成一段像专家的话，而是学会专家依赖的关键判断节点、诊断因子和决策结构。 所以我们重点看的是 contract、macro-F1、turning signal、bucket semantics，而不是 prose 是否像人。

阶段收敛

先打通可执行接口，再精修弱字段

当前阶段已经过了“找接口”和“修 contract”的部分。后续最合理的动作是冻结已验证主线， 只对最弱的 bucket 语义做定向提升。

Phase 2 不做什么

  • 不把 `reasoning_text` 拼回主 forecast target。
  • 不把“标签更准一点”当成下游真正改善的替代证据。
  • 不把 richer schema 的训练可跑通，误判为 forecast 价值已经成立。
  • 不把“会写 explanation”误解成“已经学会专家如何做决策”。
Stage Timeline

从失败分支到当前主线的阶段推进

这条主线不是凭直觉选出来的，而是通过多轮负结果与正结果共同收敛出来的。

Phase 2A

先验证接口是否真实有效

  • `track_turn`、`track_inflection`、`track_correction` 都没过主指标 gate。
  • `oracle slot_correction` 首次明确通过，成为第一条正向接口。
  • 结论：正确方向是 slot-locked、track-only、可执行接口。
Phase 2B

再验证 predicted 能否恢复 oracle 增益

  • `slot_correction v0` 有部分 gain，但 contract 失败。
  • `v1` 修好 contract 主问题，但还差 dual-track gate。
  • 加入 baseline intensity 与 scale 1.20 后，在更大 held-out confirm 上正式通过。
Phase 2C

只做窄而可执行的扩展

  • `track_core v1` 证明 generic richer schema 不是当前主线。
  • 转向 `slot_turn_correction` 这条更窄的 executable extension。
  • `v1` 同时守住 forecast-side gain 和 standalone contract，Phase 2C 收口完成。
Next Phase

进入 bucket semantic resolution

  • 固定 13 字段 contract、renderer、baseline intensity integration。
  • 不再修 turning，而是专门打最弱的 slot bucket 语义。
  • 核心瓶颈已经收窄到 `lat bucket`。
Frontier

`v3` 已完成主线升级

  • 更大 held-out full confirm 继续优于 `v1` control，已升级为当前 confirmed mainline。
  • 增益主要来自 `lon bucket`，不是原先预想中的 `lat repair`。
  • 下一步不再是补 confirm，而是判断是否已具备安全进入 RL 的条件。
Hard Evidence

Reasoning Evidence

除了直接给出预报，我们如何判断模型是否学到了“如何推理”

当前阶段并不以长文本 reasoning prose 为目标；推理学习的证据，体现在模型是否能恢复并稳定输出 专家在预报时依赖的结构化中间判断状态。

当前“学会推理”的正确定义

  • 不是“学会写一段像专家的话”。
  • 而是学会专家使用哪些证据、如何在关键信号间做判断、如何在模式分歧里取舍。
  • 因此本阶段最重要的不是 explanation prose，而是 `structured diagnostics / decision states`。
  • 长期链路应是：`input -> diagnostics -> forecast -> explanation`。

在当前实验里，推理学习如何被观测

  • Oracle pass：证明这套 decision interface 本身有 forecast 价值，不是伪标签。
  • Predicted recovery：证明模型能从同样证据中恢复这套 decision state，而不是只能吃“正确答案”。
  • Standalone contract：证明模型能稳定表达中间判断状态，而不是只能依赖后处理 fallback。
  • Field-level 指标：例如 `turning_signal`、bucket-only macro-F1，直接对应具体判断节点是否学会。

最强证据：`slot_turn_correction v3` 已不只是 forecast-side 有效

  • `json_parse_rate = 1.0000`：说明 raw diagnostic head 已稳定守住 contract。
  • `turning_signal exact_accuracy = 0.8221`，`macro-F1 = 0.8256`：说明关键转向判断仍保持稳定，没有因主线升级而塌缩。
  • `mean_macro-F1 = 0.3418`：说明模型学到了一部分结构化诊断字段，而不是完全靠 renderer 补救。
  • 与此同时 forecast 指标继续优于 `v1` control，说明这些中间判断是“可执行的”，不是装饰性标签。

仍不能夸大：我们还没有“学会全部推理”

  • `joint_exact` 依然很低，说明全字段同时命中的能力还弱。
  • 最弱部分已经收敛到 12 个 slot bucket 里的 `lat bucket semantics`。
  • `v3 full confirm` 依然说明 aggregate 指标能继续升，但增益主要来自 `lon bucket`，不是 `lat bucket`。
  • 这意味着我们已经学会了“部分结构化决策”，但还没有学会“完整、均衡的决策分解”。

为什么当前不直接讲“reasoning text 学到了多少”

因为 reasoning 在项目设计里是 Phase 3 的解释层，而不是 Phase 2 的主决策层。当前 `reasoning_only` 数据规模只有 `train 385 / val 138 / test 148`，远小于 Phase 2 各 diagnostic view 的 `3959 / 1354 / 1613`。 所以现在更稳、更可信的说法是： 我们已经证明模型能学会一部分专家的结构化判断过程，但还没有进入 explanation prose 的系统训练阶段。

Failure Archive

失败分支告诉了我们什么

这些负结果不是附属信息，而是研究收敛的重要依据：它们解释了为什么当前最合理的策略是继续收窄，而不是重新发散。

失败分支 1：`track_turn` / `track_inflection` / `track_correction`

  • `track_turn`：standalone 指标有提升，但没有转成 track 主指标改善，说明不能用 label score 替代 downstream 结论。
  • `track_inflection`：oracle 自己都没改善主指标，说明接口过于抽象，还存在 timing leakage 风险。
  • `track_correction`：比环境标签更直接，但 fixed-lead anchor 仍太间接，无法稳定缩小 track gap。

失败分支 2：`track_core v1`

  • 训练能跑通，contract 也稳定，但 richer diagnostics 没有转成更好的 forecast track gain。
  • 这说明当前阶段不能简单沿“更宽 schema、更丰富字段”继续铺开。
  • 也因此 Phase 2C 的正确方向变成了更窄但可执行的 `slot_turn_correction`，而不是 `track_core v2`。
Plan

下一步计划：进入 RL 准备阶段，但先冻结 reward 与 guardrail

当前最合理的计划不是回到 redesign；`v3 full confirm` 和 post-confirm semantic audit 已经说明主线可以稳定进入 RL 准备阶段。 但更稳妥的顺序仍然是：先把 reward 与 guardrail 冻结，再正式启动 RL 训练。

Step 1：semantic audit 已完成

  • 当前 verdict 已明确：`v3` 的 held-out gain 主要来自 `lon semantics`。
  • `lat bucket` 仍是最弱语义轴，但不是当前 gain 的主要解释变量。
  • 因此新的 `lat-only targeted SFT cleanup` 不再作为 RL 前的必经 blocker。

Step 2：冻结 RL objective 与 guardrail

  • 如果进入 RL，优化对象应是 integrated forecast behavior，而不是 raw diagnostic head。
  • 主 reward 必须直接绑定 `track_error_km` 与 `mean_track_diff_vs_official_km`，不能用 standalone proxy 替代。
  • `intensity_error_kt`、`strict_parseable_rate`、`slot_time_match_rate_vs_official` 更适合作为 guardrail。

Step 3：评估口径保持严格

  • 继续同时看两类指标：forecast-side 主指标，以及 standalone diagnostic contract / field-level 指标。
  • 不允许只凭 `reward_mean`、`coverage` 或 `mean_macro-F1` 上升就宣称 RL 值得启动。
  • 继续单独跟踪 `bucket-only mean macro-F1`，避免 `turning_signal` 掩盖真正短板。

Step 4：Phase 3 先不提前启动

  • reasoning/explanation 仍应作为后续独立阶段，而不是现在与 Phase 2 主线混训。
  • 当前更稳的叙述是：我们先把结构化 decision state 学稳，并把 RL reward 设计收口，再考虑 explanation 层。
  • 这能避免为了长文本解释，破坏已经验证通过的 forecast 主链路。

下一步的核心表述

“当前已经可以开始准备 RL。更准确地说：我们不会再把 `lat-only cleanup` 作为 RL 前的 mandatory blocker， 而是直接以当前 confirmed mainline 为基础，先冻结只绑定 dual-track gate 的 RL objective 与 guardrail， 然后再启动 RL 训练。”
