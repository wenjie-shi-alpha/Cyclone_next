**《大语言模型预报任务中的数据泄露防治规范（Draft v1.0）》**

**适用范围**
本规范适用于将大语言模型作为“预报员”进行微调、指令微调、偏好优化或评测的任务，尤其适用于热带气旋、强对流、空气污染过程、灾害影响等具有明确时间演化和历史事件标签的预报问题。其目标不是证明模型“绝对不知道历史”，而是把任务设计成尽可能接近真实业务预报场景，并通过审计实验量化剩余污染风险。现有关于 LLM 污染的综述和检测研究都指出，训练语料不透明、模型预训练记忆和评测样本公开传播，使“零污染”通常难以严格证明，因此研究重点应放在**预防 + 隔离 + 审计 + 报告**四个环节。([arXiv][1])

**一、核心问题定义**
在本任务中，数据泄露主要包括四类。第一类是**时间泄露**：输入中混入了预报发布时点之后才会知道的信息。第二类是**事件泄露**：同一气旋或同一事件的不同时间片同时出现在训练集和测试集中，使模型通过事件记忆而非规律推理完成预测。第三类是**身份泄露**：样本包含台风名、精确日期、地点、灾情描述等线索，使模型能直接把样本与历史著名事件对上号。第四类是**预训练污染**：底座模型在大规模预训练阶段已经接触过相关事件文本、总结报告、新闻或公开数据，导致测试时调用记忆而不是依据给定证据推理。污染研究普遍认为，这几类问题都会导致性能被高估，并削弱评测对真实泛化能力的解释力。([arXiv][1])

**二、总原则**
所有训练、验证、测试样本必须遵循一个唯一原则：**模型在样本输入端只能看到该预报时点此前真实可获得的信息，且测试事件在训练阶段不得以任何形式出现。**
换句话说，任务应被构造成一个 operational forecasting setting，而不是一个“已知答案后的历史复盘问答任务”。近年来面向预报的 LLM 评测工作，已经明确采用“冻结当时上下文快照（frozen context snapshots）”来减少 temporal leakage；而热带气旋预测基准 TCBench 也明确采用按年份划分风暴事件来防止 leakage。([OpenReview][2])

**三、必须执行的数据切分规则**

1. **严禁随机切分样本。** 不允许将同一风暴、同一事件链、同一过程的不同时间片随机分配到 train / val / test。
2. **必须按事件切分。** 同一 storm ID 只能属于一个数据子集。
3. **必须按时间切分。** 推荐采用“早年训练、次新年份验证、最新年份测试”的方式，确保测试集是严格未来数据。TCBench 的做法就是按风暴发生年份划分，并明确说明这样做是为了防止泄露；其建议配置为 2017–2020 训练、2021–2022 验证、2023 测试。
4. **测试集必须长期冻结。** 一旦确定测试年和测试事件，不得在提示工程、误差分析、人工规则设计和数据清洗阶段反复利用测试集做迭代优化。
5. **若样本间存在强关联背景，应加做更严版本。** 例如可增加 basin 留出、season 留出或极端路径类型留出，用于评估更强泛化。([arXiv][3])

**四、输入构造规则：只允许“预报时点前可见信息”**
每个训练或测试样本都必须绑定一个明确的 forecast issuance time。输入中只能包含这一时点及之前可见的信息，例如过去若干时次的位置、强度、环境场特征、卫星摘要、数值模式摘要或人工整理的截至该时点的结构化观测。
以下信息必须禁止进入输入：

* best-track 的未来轨迹与未来强度
* 事后回顾性总结
* 登陆结果、伤亡结果、最终等级
* 包含“后来证明”“最终在某地登陆”等后验表达的自由文本
* 从未来窗口反推得到的派生特征
  近期预报评测研究使用 frozen context snapshots，核心就是把问题严格限制在“当时看得到什么，就只允许模型基于什么回答”。这条原则在你的任务里应作为硬约束，而不是建议项。([OpenReview][2])

**五、匿名化与去标识化规则**
即使完成了时间切分，模型仍可能因为样本中包含强标识符而“认出”历史事件。因此，训练集、验证集和测试集都应建立匿名化流水线：

* 台风名替换为匿名事件 ID
* 精确公历日期可改写为相对时序索引或 season-relative 表达
* 对地点进行层级泛化，避免直接暴露极具辨识度的地名组合
* 删除新闻式描述、灾情报道式语言、百科式摘要
* 对自由文本做模板化重写，只保留预报所需事实
  VCBench 的做法表明，匿名化加对抗性重识别测试可以显著降低重识别风险；RAT-Bench 进一步指出，即使最好的匿名化工具也并不完美，尤其当间接标识符仍然保留时，重识别风险依然存在。因此，你的目标不应是“做一次脱敏就结束”，而应是“匿名化后继续做重识别攻击测试”。([arXiv][4])

**六、训练语料治理规则**

1. **底座优先选择可追溯模型。** 能使用公开模型版本、公开训练说明的底座，就不要优先依赖完全黑箱的闭源底座。
2. **微调语料不得包含测试事件的公开复盘材料。** 包括论文、百科、新闻专题、赛后总结、案例教学资料。
3. **训练前做去重与近重复扫描。** 对 train 与 val/test 之间进行字符串重合、n-gram 重合、语义近邻检索和模板重写后再检索。
4. **自由文本训练样本必须重写。** 不能直接把公开历史公报、新闻和复盘文本原样喂给模型。
5. **标签与输入生成链路必须隔离。** 负责生成标签的程序不得把未来轨迹信息回写到输入摘要中。
   污染综述普遍将 mitigation 分为更新型、重写型和预防型三大类；对你这个任务而言，最关键的是“预防型切分”和“重写型去标识化”同时使用。([arXiv][1])

**七、评测设计规则：必须做多层测试，而不是只报一个分数**
正式实验至少应包含以下四组测试：
**A. 主测试集**：未见年份 + 未见风暴。
**B. 匿名测试集**：在主测试集基础上去掉风暴名、日期、地点等识别线索。
**C. 结构化-only 测试集**：只给结构化气象输入，不给任何容易唤起历史记忆的自然语言线索。
**D. 干扰测试集**：对名称、日期表述、地名粒度做扰动，看结果是否异常波动。
如果模型在 A 上很高、在 B/C 上明显下降，通常说明它部分依赖事件识别而非真正基于证据推理。这个思路与当前 LLM 污染评测和预报回测的设计原则是一致的。([OpenReview][2])

**八、污染审计实验：必须与主结果一起报告**
主实验之外，必须增加污染审计。建议至少包括以下五项：

1. **重合度扫描**：检查训练集与测试集在文本、模板、字段组合上的直接或近似重合。
2. **线索消融**：逐步去掉名字、日期、地点、灾情描述，观察性能变化。
3. **重识别攻击**：给一个外部模型或规则系统，只基于匿名文本去猜事件身份，衡量匿名化是否足够。
4. **对抗重写复测**：将测试样本做同义重写、模板重写，再测性能。
5. **模型级污染检测**：使用 CoDeC、KDS、CDD 等方法对底座或微调后模型做污染量化，但必须明确说明这些方法只能提供证据，不能替代严格的数据协议。
   CoDeC 这类新方法已经展示出较强的 seen/unseen 区分能力；但 COLING 2025 的研究也指出，现代 LLM 的污染检测方法仍存在局限、一致性问题和 oracle challenge，所以审计方法要“多证据并用”，不能只依赖单一检测器。([OpenReview][5])

**九、论文与报告中的最低披露要求**
凡是发表结果或内部汇报，必须明确披露：

* 底座模型名称、版本、可获得的训练信息范围
* 训练/验证/测试的时间范围
* 是否按 storm/event 独占切分
* 输入是否严格限定在预报时点前可见信息
* 是否进行了匿名化
* 是否做了污染审计，以及采用了哪些方法
* 哪些风险仍然无法完全排除
  这不是形式问题，而是结果可信度的一部分。污染研究之所以反复强调 protocol，就是因为很多“高分”本质上来自未披露的泄露路径。([arXiv][1])

**十、推荐执行流程**
在工程实现上，推荐采用以下顺序：
先确定测试年份与测试风暴列表，并永久冻结；
再基于事件级规则切出训练和验证集；
再为每个样本定义 issuance time，仅生成此前可见输入；
再进行匿名化与重写；
然后进行 train/val/test 去重与近重复扫描；
之后才开始微调；
微调完成后，先跑匿名测试与结构化-only 测试，再跑主测试；
最后统一输出主结果、消融结果、污染审计和风险声明。
这个顺序的关键点在于：**先锁测试，再做数据与模型开发**，避免开发过程被测试集反向污染。([arXiv][3])

**十一、可直接执行的红线清单**
以下行为视为不合规：

* 随机切分同一风暴的不同时间片
* 使用包含未来结果的文本摘要作为输入
* 在测试集上反复调 prompt、改模板、做规则修正
* 训练语料中包含测试事件的新闻、百科或总结
* 在样本中保留足以唯一定位历史个案的标识符
* 只报告单一测试分数，不报告匿名化与消融结果
* 用一种污染检测方法“证明没有污染”
  由于匿名化和检测方法本身都有局限，真正的防泄露依赖的是**制度化的数据协议**，而不是事后补救。([arXiv][6])

**十二、给 AI 的执行指令**
当 AI 参与数据集构建、清洗、训练样本生成或评测时，必须遵守以下指令：
“不得将 forecast issuance time 之后的任何信息写入输入。不得在不同数据子集之间复用同一 storm/event。必须移除或改写风暴名、精确日期、强标识地点和事后总结。生成样本后必须进行去重、近重复检索和泄露检查。若无法确认某字段是否属于未来信息，则默认删除。若匿名化后仍可通过文本推断事件身份，则样本不合格。”
这条 AI 指令的目的，是把防泄露从“人工提醒”变成数据生产流程中的自动约束。其总体方向与当前关于预报回测、匿名化和污染检测的研究实践一致。([OpenReview][2])
